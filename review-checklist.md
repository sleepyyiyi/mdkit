# AI Code Review 四象限清单（Go 后端）

> 给 AI 跑 Review 时把这份清单贴进 prompt，让它逐项过。
> 人工审核也按这份兜底，任何一项没看过不许 merge。完整约定见 [CLAUDE.md](./CLAUDE.md)。

## 用法（喂给 AI 的 prompt 模板）

```
请对以下 diff 按"四象限清单"逐项 review。
每一项给出：是否命中 / 位置（文件:行号）/ 严重等级（阻塞/警告/建议）/ 修复建议。
最后给一句"是否建议 merge"。
--- 清单 ---
<粘下面四个象限>
--- diff ---
<贴 diff 或 @相关文件>
```

## 象限 ①｜性能（Performance）
- [ ] 循环内是否有重 IO / 重复 query（应批处理或预取）
- [ ] `resp.Body` / `rows` / 文件句柄是否 `defer Close()`，有无泄漏
- [ ] goroutine 是否有退出路径，是否会无界增长；channel 是否可能阻塞泄漏
- [ ] 是否用了 `context` 控制超时/取消，长调用是否可中断
- [ ] 大 slice/map 是否预分配容量；热点路径是否有不必要的内存分配
- [ ] 锁粒度是否合理，临界区是否过大

## 象限 ②｜安全（Security）
- [ ] SQL 是否参数化（**禁止**字符串拼接 SQL）
- [ ] 入参是否校验/限长，路径/命令是否有注入风险
- [ ] 鉴权/权限校验是否覆盖所有需要保护的接口
- [ ] 错误响应是否泄露内部堆栈 / SQL / 内部结构
- [ ] 密钥/Token 是否走配置或环境变量，**不入日志、不入仓库**
- [ ] 外部请求是否校验来源 / 超时；反序列化是否限制大小（防 DoS）

## 象限 ③｜正确性 / 错误处理（Correctness）
- [ ] 每个 `err` 是否被处理，**无吞错**（`_ = err` 要有理由）
- [ ] 错误是否用 `%w` 包装并保留上下文
- [ ] 并发访问的共享状态是否有保护（`go test -race` 是否过）
- [ ] 边界：空输入 / nil / 零值 / 超时分支是否都覆盖
- [ ] 资源在所有返回路径（含 error）上是否都释放

## 象限 ④｜可维护性（Complexity）
- [ ] handler 是否混入了业务逻辑（应在 service）
- [ ] 依赖方向是否单向（无反向依赖、无包级全局可变状态）
- [ ] 函数过长 / 圈复杂度过高是否需要拆
- [ ] 导出符号是否有 doc comment；命名是否符合 Go 习惯
- [ ] 是否引入"看起来通用但只一处用"的抽象
- [ ] 测试覆盖：核心逻辑 / 异常 / 边界

## 人工最终审核（Merge 前必看）
- [ ] 核心业务逻辑符合需求
- [ ] 异常路径：失败、超时、并发都覆盖
- [ ] 性能影响：高 QPS / 大数据量场景
- [ ] 安全风险：见象限 ②
- [ ] 可维护性：3 个月后能不能看懂

任何一项答不上来 → 退回重做或继续补测，不许 merge。
